#### 黑盒测试

1. 黑盒测试：

   - 基于规格说明书的测试
   - 从用户观点出发的测试
   - 测试人员一般把被测程序当作一个**黑盒子**

2. 黑盒测试一般能检查出什么错误？

   - 不正确或遗漏的功能
   - 接口、界面错误
   - 性能错误
   - 数据结构或外部数据访问错误
   - 初始化或终止条件错误
   - ......

3. 黑盒测试有哪些方法？

   - 等价类划分法

     有效等价类&无效等价类

   - 边界值分析法

     设计边界值附近的测试用例

   - 因果图法

     要了解一下怎么画因果图、因果图转决策表

     这篇说得挺详细👉[软件测试：因果图法](https://blog.csdn.net/xionghuixionghui/article/details/68216564)

   - 场景法

     基本流&备选流：基本流是正确流程，备选流就是用户中间奇奇怪怪的错误操作

   - 正交实验设计法

     n个控件，每个有m个选择，就是n因素m水平

     列出所有用例后，可以用正交排列表减少测试用例的个数

     具体正交排列表长啥样？指路👉[正交排列法](https://www.cnblogs.com/KalosOwen/p/8253933.html)

   - 判定表驱动分析法

     条件桩、动作桩、条件项、动作项

     步骤：

     1. 确定规则的个数
     2. 列出所有的条件桩和动作桩
     3. 填入条件项
     4. 填入动作桩和动作项
     5. 化简判定表

     具体案例见👉[判定表驱动分析方法](https://blog.csdn.net/xiaofuruo/article/details/17711093)

   - 错误推测法

     竟然就是按经验猜（推测）哪里出问题，然后有针对性地设计测试用例🙈

#### 白盒测试

- 语句覆盖

  让每条代码都用上

- 判定覆盖

  保证每个判断的分支都至少经历一次

- 条件覆盖

  让每个条件的所有可能结果至少出现一次（但未必能覆盖全部分支）

- 判定条件覆盖

  判定覆盖和条件覆盖的合体，都要满足

- 条件组合覆盖

  让各种条件的组合都至少经历一次

- 路径覆盖

  每条可能执行到的路径至少执行一次

- 页面响应的整个过程[《What really happens when you navigate to a URL》](http://igoro.com/archive/what-really-happens-when-you-navigate-to-a-url/)

  1. 输入地址
  2. 浏览器查找域名的 `IP` 地址
     这一步包括 `DNS` 具体的查找过程，包括：浏览器缓存->系统缓存->路由器缓存...
  3. 浏览器向 `web` 服务器发送一个 `HTTP` 请求
  4. 服务器的永久重定向响应（从 `http://example.com` 到 `http://www.example.com`）
  5. 浏览器跟踪重定向地址
  6. 服务器处理请求
  7. 服务器返回一个 `HTTP` 响应
  8. 浏览器显示 `HTML`
  9. 浏览器发送请求获取嵌入在 `HTML` 中的资源（如图片、音频、视频、`CSS`、`JS`等等）
  10. 浏览器发送异步请求

- 三次握手可以是两次吗

  三次握手：

  1. 第一次握手，发送SYN报文，传达信息：“你好，我想建立连接”；
  2. 第二次握手，回传SYN+ACK报文，传达信息：“好的，可以建立链接”；
  3. 第三次握手，回传ACK报文，传到信息：“好的，我知道了，那我们连接”。然后就建立连接了

  两次握手是说去掉第三次握手，如果发送SYN的一方不想连接了，也不会有反馈，另一方却一直在等待，浪费了时间

- HTTP状态码

  - 1打头：通知
  - 2打头：操作成功
  - 3打头：重定向（客户端需要做些额外工作才能得到所需要的资源）
  - 4打头：客户端错误
  - 5打头：服务端错误

  200 OK：请求成功，信息在返回的响应报文中。

  301 Moved Permanently：请求的对象已经被永久转移了，新的URL定义在响应报文的Location：首部行中。客户软件将自动获取新的URL。

  400 Bad Request：一个通用差错代码，指示该请求不能被服务器理解。

  404 Not Found：被请求的文档不在服务器上

  505 HTTP Version Not Supported：服务器不支持请求报文使用的HTTP协议版本

- http和https区别

  https比http安全（s是security），原因是加上了SSL协议。SSL依靠证书来验证服务器身份，并为浏览器和服务器之间的通信加密

  1. https协议需要到ca申请证书，一般免费证书较少，因而需要一定费用。
  2. http是超文本传输协议，信息是明文传输，https则是具有安全性的ssl加密传输协议。
  3. http和https使用的是完全不同的连接方式，用的端口也不一样，前者是80，后者是443。
  4. http的连接很简单，是无状态的；HTTPS协议是由SSL+HTTP协议构建的可进行加密传输、身份认证的网络协议，比http协议安全。

- GET和POST的区别

  | 请求方式       | GET                                | POST                         |
  | -------------- | ---------------------------------- | ---------------------------- |
  | 参数位置       | URL的query中                       | 一般在content中              |
  | 参数大小       | 受限于浏览器url大小，一般不超过32K | 1G                           |
  | 服务器数据接收 | 接收1次                            | 根据数据大小，可分多次接收   |
  | 适用场景       | 从服务器端获取数据，不做增删改     | 向服务器提交数据，可做增删改 |
  | 安全性         | 参数携带在URL中，安全性低          | 相对于GET，安全性更高        |

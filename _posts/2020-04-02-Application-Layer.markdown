## 1. 应用层协议原理

1. 两种应用程序体系结构：**客户-服务器**、对等（**P2P**）
* 客户-服务器：
  * 客户之间不直接通信；服务器具有固定的、周知的地址（即IP地址）
  * e.g. Web、FTP、Telnet、电子邮件
  * 缺陷：单独一台服务器主机可能跟不上所有客户请求->数据中心（需要供电和维护；服务商必须支付发送和接收到达/来自数据中心数据的互联和带宽费用）
* P2P：
  * 应用程序在间断连接的主机对之间直接通信（对等方）
  * e.g. 迅雷、Skype、IPTV
  * 优点：自扩展性
  * 挑战：ISP友好（现在带宽下载>上传）、安全性（P2P高度分布和开放）、激励（P2P需要用户自愿向应用提供带宽、存储和计算资源）
2. 进程通信
* 进程：
   进行通信的实际上是**进程**而不是程序。在两个不同端系统上的进程，通过跨越计算机网络交换**报文**message而相互通信。
   
   定义发起通信的进程是客户，等待联系的进程是服务器。
   
   P2P中的进程能够既是客户又是服务器。

* 接口：

  进程通过**套接字**socket向网络发送报文/从网络接收报文。

  套接字是应用程序和网络之间的**应用程序编程接口**API

  应用程序开发者可以控制套接字在应用层端的一切，但对它在传输层内几乎没有控制权（仅能选择传输层协议；也许能设定几个传输层参数）

* 进程寻址：

  IP地址+端口号

  Web服务器端口号80，邮件服务器（用SMTP协议）端口号25

* 运输服务：

  * 可靠数据传输

    TCP提供可靠数据传输：确保数据正确、完全地传到另一端，用于Email、文件传输、Web文档传输、金融应用等

    UDP不提供可靠数据传输，但是快，应用于网络电话等（能承受一定量的数据丢失）

  * 吞吐量

    网络电话等应用是**带宽敏感的应用**（需要维持同样的编码速率）

    Email、文件传输以及Web传送属于**弹性应用**

    当然，吞吐量越大越好

  * 定时

    指对数据交付的时延有时间限制（如网络电话、互动游戏）

  * 安全性

    传输协议对数据进行加密，防止在传输过程中被嗅探到

| 应用              | 数据丢失 | 带宽                   | 时间敏感  |
| ----------------- | -------- | ---------------------- | --------- |
| 文件传输          | 不能     | 弹性                   | 否        |
| 电子邮件          | 不能     | 弹性                   | 否        |
| Web文档           | 不能     | 弹性                   | 否        |
| 网络电话/视频会议 | 容忍     | 音频带宽较低，视频略高 | 是，100ms |
| 存储音频/视频     | 容忍     | 同上                   | 是，几秒  |
| 交互式游戏        | 容忍     | 较低的带宽             | 是，100ms |
| 即时讯息          | 不能     | 弹性                   | 是和否    |

* 因特网传输层提供的服务

  * TCP：面向连接服务+可靠数据传输服务，还具有拥塞控制机制（即当发送和接收方之间的网络出现拥塞时，TCP会抑制发送进程，达到总体公平共享网络带宽的目的）

  * UDP：仅提供最小服务（无握手过程、不可靠数据传送、可能乱序到达、无拥塞控制机制）

  * SSL：即安全套接字层，是一种对TCP的加强，强化在应用层上，实现对明文数据加密

  * 因特网运输协议不提供对吞吐量和定时的保证

  * 因特网电话由UDP或TCP支撑（TCP作为UCP失败时的备份），其他应用大部分使用TCP

* 应用层协议

  定义了：

  * 交换的报文类型
  * 各种报文类型的语法
  * 字段的语义
  * 一个进程何时发送报文、如何发送报文、如何响应报文

  应用层协议只是应用的一部分

## 2. Web和HTTP

1. HTTP概况

   HTTP是**无状态协议**stateless protocol，即服务器不存储任何客户的状态信息，仿佛完全忘记不久之前所做过的事一样

2. 非持续链接和持续链接
   * 非持续连接的情况下，从服务器向客户传送一个Web页面的步骤：
     1. HTTP客户进程在端口号80发起一个到服务器www.sjtu.edu.cn的TCP连接
     2. HTTP客户经它的套接字向服务器发送一个HTTP请求报文，里面包含了路径名/someDepartment/home.index
     3. HTTP服务器进程经它的套接字接收该请求报文，从其存储器中检索出对象www.sjtu.edu.cn/someDepartment/home.index，在一个HTTP响应报文中封装对象，并通过其套接字向客户发送响应报文
     4. **HTTP服务器进程通知TCP断开该TCP连接**（TCP在确认客户已经完整地收到响应报文后，才会实际中断连接）
     5. HTTP客户接收响应报文，TCP连接关闭。该报文指出封装的对象是一个HTML文件。客户从响应报文中提取出该文件，检查该HTML文件，得到对10个JPEG图形的引用
     6. 对每个引用的JPEG图形对象重复前四个步骤
